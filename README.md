# Bedrock Retail Support Intelligence Pipeline

An AWS-native data engineering pipeline that uses Amazon Bedrock to transform unstructured customer support ticket text into structured, analytics-ready datasets. Raw retail operations data flows through a medallion architecture — standardization, AI-powered enrichment, semantic similarity analysis, and gold-layer aggregation — culminating in an automated daily operations brief generated by a foundation model.

The full pipeline is orchestrated end to end with AWS Step Functions: a single execution runs two Glue jobs, four Lambda functions, and three Bedrock model interactions in sequence.

> **Not a RAG system.** This is deliberately not Retrieval-Augmented Generation: no vector database, no retrieval step grounding the prompts, no Bedrock Knowledge Base. Embeddings are used solely for semantic similarity analysis between historical tickets.

## How Amazon Bedrock Is Used

Bedrock is integrated at three distinct stages, each with a different job.

### Stage 1 — Ticket Enrichment (text model)

- **Model:** `amazon.nova-lite-v1:0` · **API:** Converse · **Lambda:** `ticket_enrichment`

Each ticket's raw customer message and agent notes are sent to the text model, which returns structured JSON:

| Field | Description | Example |
|---|---|---|
| `issue_category` | Classified issue type | `DELIVERY_DELAY` |
| `sentiment` | Customer sentiment | `NEGATIVE` |
| `urgency` | Priority level | `HIGH` |
| `requires_human_followup` | Whether manual intervention is needed | `true` |
| `root_cause_hint` | Likely cause | `Transit delay due to carrier backlog` |
| `pii_redacted_message` | Message with phone/email masked | `My order O10001 is late...` |
| `customer_summary` | One-sentence summary | `Customer reports a delivery delay of over 24 hours.` |
| `next_best_action` | Recommended operations action | `Escalate to carrier for expedited delivery.` |

This enrichment output feeds every downstream stage.

### Stage 2 — Ticket Embeddings (embedding model)

- **Model:** `amazon.titan-embed-text-v2:0` · **API:** InvokeModel · **Lambda:** `ticket_embeddings`

A composite string (issue_category + customer_summary + pii_redacted_message) is embedded into a vector. Tickets about the same underlying problem land close together in vector space even when the wording differs — e.g. *"my package never arrived"* and *"the courier marked it delivered but I never received it"* score high cosine similarity. Vectors are used only for similarity, never for retrieval.

### Stage 3 — Daily Operations Brief (text model)

- **Model:** `amazon.nova-lite-v1:0` · **API:** Converse · **Lambda:** `daily_ops_brief`

After the gold tables are built, this Lambda pulls aggregated metrics from Athena (top issue categories, city hotspots) and sends them to the text model, which writes a business-facing brief: a headline, a paragraph on dominant issue patterns, a paragraph on what to prioritize, and the highest-risk category called out explicitly.

## Source Data

Synthetic but internally consistent datasets live in `data/raw/`:

| Dataset | Format | Rows | Description |
|---|---|---|---|
| `orders.csv` | CSV | 180 | Orders with product, city, amount, payment status |
| `shipments.csv` | CSV | 180 | Shipments linked to orders, including delay_hours |
| `returns.csv` | CSV | 52 | Return requests with refund amounts |
| `products.csv` | CSV | 20 | Product catalog with categories and prices |
| `support_tickets.jsonl` | JSONL | 92 | Free-text customer messages and agent notes |

Relationships are realistic by design: delayed shipments align with delivery-delay complaints, damaged-item tickets correspond to damage-reason returns, refund-delay complaints follow return approvals, and so on.

## Pipeline Components

### Glue — `bronze_to_silver_standardize`
Reads raw S3 data and produces cleaned, type-safe silver datasets: numeric casts, timestamp parsing, uppercase normalization, and two derived shipment flags (`delivery_delayed_flag` for delays > 24h, `delivery_exception_flag` for DELAYED / DELIVERED_DAMAGED status). Structured tables are written as CSV; support tickets as JSONL (free text breaks CSV).

### Glue — `build_gold_support_mart`
Runs after the three Bedrock Lambda stages. Joins silver operational data with enrichment and similarity outputs into three gold tables:

| Table | Description | Consumed by |
|---|---|---|
| `order_support_360` | One row per order with enrichment, similarity, and derived flags | Athena queries |
| `issue_kpis` | Aggregated metrics per issue_category | `daily_ops_brief` |
| `city_issue_hotspots` | Aggregated metrics per (city, issue_category) | `daily_ops_brief` |

### Lambda functions
- **`ticket_enrichment`** — silver tickets → Bedrock text model → structured JSON.
- **`ticket_embeddings`** — enriched tickets → Bedrock embedding model → vectors.
- **`ticket_similarity`** — vectors → cosine similarity → top-N matches per ticket. No model call; pure math.
- **`daily_ops_brief`** — Athena metrics → Bedrock text model → Markdown brief. Auto-creates the Athena external tables if missing.

### Athena queries
Four analytical queries in `athena/queries/`: issue volume by category, city hotspots, high-risk orders (prioritized by urgency), and cross-category similarity patterns.

## AWS Services Used

S3 (medallion storage), AWS Glue (PySpark standardization + gold joins), AWS Lambda (Bedrock orchestration + similarity math), Amazon Bedrock (Nova Lite text, Titan V2 embeddings), Amazon Athena (SQL over gold tables), AWS Step Functions (end-to-end orchestration), IAM (least-privilege roles).

## Setup & Execution

### Prerequisites

- An AWS account with access to **Amazon Bedrock** in your chosen region.
- Bedrock model access enabled for **Amazon Nova Lite** and **Titan Text Embeddings V2**.
- Region used in this project: **us-east-1**  <!-- FILL IN if you used a different region -->

### 1. Create the S3 bucket and upload raw data

```bash
# FILL IN: your bucket name
export PIPELINE_BUCKET=<your-bucket-name>

aws s3 mb s3://$PIPELINE_BUCKET --region us-east-1
aws s3 cp data/raw/ s3://$PIPELINE_BUCKET/raw/ --recursive
```

### 2. Create the IAM roles

- A **Glue role** with read/write on the pipeline bucket (`s3:GetObject`, `s3:PutObject`, `s3:DeleteObject`, `s3:ListBucket`) plus the standard Glue service permissions.
- A **Lambda role** with bucket read/write, `bedrock:InvokeModel`, and Athena + Glue Data Catalog permissions for the brief stage.

### 3. Create the two Glue jobs

Upload the scripts from `glue/scripts/` and set these job parameters:

| Key | Value |
|---|---|
| `--PIPELINE_BUCKET` | `<your-bucket-name>` |
| `--RAW_PREFIX` | `raw` |
| `--SILVER_PREFIX` | `silver` |

(The gold job uses `--SILVER_PREFIX`, `--BEDROCK_PREFIX`, and `--GOLD_PREFIX`.)

### 4. Deploy the four Lambda functions

Environment variables per function:

**ticket_enrichment**

| Variable | Value |
|---|---|
| `PIPELINE_BUCKET` | `<your-bucket-name>` |
| `SILVER_PREFIX` | `silver` |
| `BEDROCK_PREFIX` | `bedrock` |
| `TICKET_ENRICHMENT_MODEL_ID` | `us.amazon.nova-lite-v1:0` |

**ticket_embeddings**

| Variable | Value |
|---|---|
| `PIPELINE_BUCKET` | `<your-bucket-name>` |
| `BEDROCK_PREFIX` | `bedrock` |
| `TICKET_EMBEDDING_MODEL_ID` | `amazon.titan-embed-text-v2:0` |

**ticket_similarity**

| Variable | Value |
|---|---|
| `PIPELINE_BUCKET` | `<your-bucket-name>` |
| `BEDROCK_PREFIX` | `bedrock` |
| `TOP_MATCHES_PER_TICKET` | `3` |

**daily_ops_brief**

| Variable | Value |
|---|---|
| `PIPELINE_BUCKET` | `<your-bucket-name>` |
| `BEDROCK_PREFIX` | `bedrock` |
| `GOLD_PREFIX` | `gold` |
| `ATHENA_DATABASE` | `retail_support_intelligence` |
| `ATHENA_WORKGROUP` | `primary` |
| `ATHENA_RESULTS_PREFIX` | `athena-results` |
| `OPS_BRIEF_MODEL_ID` | `us.amazon.nova-lite-v1:0` |

> **Model ID note:** in most regions (including us-east-1), Nova Lite must be referenced through its cross-region inference profile ID `us.amazon.nova-lite-v1:0`, not the bare `amazon.nova-lite-v1:0`. The bare ID returns `ValidationException: The provided model identifier is invalid`.

### 5. Configure Athena

```sql
CREATE DATABASE IF NOT EXISTS retail_support_intelligence;
```

Set the Athena query results location to `s3://<your-bucket-name>/athena-results/`. The gold and similarity external tables are auto-created by `daily_ops_brief` (reference DDL also in `athena/ddl/`).

### 6. Run it

**Option A — Step Functions (recommended).** Replace the placeholder ARNs/job names in `step-functions/retail_support_intelligence_pipeline.asl.json`, create the state machine, attach an execution role that can start Glue jobs and invoke the Lambdas, and start one execution. It runs all six stages in order.

**Option B — manually, in order:**

1. Glue `bronze_to_silver_standardize` → verify `silver/*` populated
2. Lambda `ticket_enrichment` → verify `bedrock/ticket_enrichment/ticket_enrichment.jsonl`
3. Lambda `ticket_embeddings` → verify `bedrock/ticket_embeddings/ticket_embeddings.jsonl`
4. Lambda `ticket_similarity` → verify `bedrock/ticket_similarity/ticket_similarity.jsonl`
5. Glue `build_gold_support_mart` → verify `gold/order_support_360/`, `gold/issue_kpis/`, `gold/city_issue_hotspots/`
6. Lambda `daily_ops_brief` → verify `gold/daily_ops_brief/daily_operations_brief.md`
